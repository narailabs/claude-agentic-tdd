# Eval 5: Ambiguous Spec — Summary

This eval is designed to stress-test the design gate with a maximally vague specification. The user says "I need something that handles payments. like credit cards and stuff. also refunds maybe? idk the details yet" — which is informal, ambiguous, and lacking in technical specificity. The design gate triggers on multiple conditions: the spec mentions distinct features (payments, refunds), involves external integrations (payment processors), and is deeply ambiguous about data flow, error handling, and scope.

The critical behavior being tested is the MANDATORY trade-off proposal step. The skill specification requires that "every spec complex enough to trigger the design gate has at least one meaningful trade-off" and that the skill must "present 2-3 options with pros/cons for at least one key architectural decision." In this eval, two trade-offs are surfaced: payment processing architecture (direct integration vs. abstraction layer vs. queue-based) and refund handling strategy (synchronous vs. async queue). The clarifying questions (asked one at a time, not as a batch) transform the vague spec into concrete requirements: Visa/Mastercard only, Stripe integration, full and partial refunds. The design summary captures all decisions and explicitly lists what is out of scope. Only after user approval does decomposition begin — the HARD GATE is enforced.

The dependency graph is linear (refund-service depends on payment-service), so eager dispatch runs payment-service first, then refund-service. Both units are classified as Architecture complexity due to external API integration and security concerns, meaning Test Writers and reviewers use opus/high while Code Writers use sonnet/high. The adversarial review is especially important here because Stripe mocking is involved — the reviewer must check anti-pattern #3 (mocking without understanding side effects) and verify that mock fidelity captures real Stripe error shapes. This eval validates that the skill can transform a vague, informal user request into a rigorous, well-designed implementation through the design gate process.
