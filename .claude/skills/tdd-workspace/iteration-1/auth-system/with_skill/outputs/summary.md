# Eval 2: Auth System — Summary

This eval exercises the full complexity of the TDD skill: the design gate triggers due to 3+ features and external integrations, mandatory trade-off proposals are generated (JWT strategy, hashing algorithm), and the design summary is carried through to all downstream agents. The spec decomposes into three work units with a dependency chain (user-registration is the root, user-login and password-reset both depend on it), which forces the eager dispatch model to execute in two waves rather than all at once.

The design gate is the critical gating step. Three clarifying questions are asked one at a time (not dumped as a questionnaire), covering token expiry, email verification, and lockout behavior. Two trade-off proposals are presented with pros/cons (JWT vs sessions vs JWT+refresh; bcrypt vs argon2). Only after user approval does decomposition begin. The design summary is stored in state and passed to every Test Writer and Spec Compliance Reviewer, ensuring downstream agents understand the architectural decisions. Complexity assessment classifies user-registration as Integration (sonnet/high for all agents) and user-login/password-reset as Architecture (opus/high for Test Writers and reviewers, sonnet/high for Code Writers and code quality reviewers). This demonstrates the model/effort assignment system working with a mixed-complexity set.

Eager dispatch is demonstrated clearly: user-registration runs first (the only unit with no dependencies), and the moment it completes, both user-login and password-reset are immediately added to the ready queue and dispatched in parallel (2 of 4 max concurrent slots). This is faster than traditional batching because neither dependent unit waits for the other. The three-stage review ordering is enforced for all three units: spec compliance must pass before adversarial review, which must pass before code quality review. The adversarial reviewers (opus/high) are expected to probe deeply on security-sensitive code: JWT secret management, lockout bypass, token predictability, email enumeration. The final integration check in Phase 5 verifies cross-unit compatibility.
