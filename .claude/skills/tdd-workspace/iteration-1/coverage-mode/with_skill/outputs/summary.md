# Eval 4: Coverage Mode — Summary

This eval exercises Mode 2 (existing codebase), which has the most unique adaptations to the standard TDD pipeline. The skill must characterize existing code behavior rather than implement new features, which inverts the normal RED verification logic. Instead of requiring tests to fail (which would be wrong for characterization), the skill uses the hide-and-restore protocol: temporarily rename the implementation file, verify tests fail with import errors (proving they depend on real code), restore the file, then verify tests pass (proving characterization is accurate). This two-step check catches both tautological tests (that pass without the implementation) and inaccurate characterization (tests that don't match actual behavior).

The Code Writer is a no-op for all three units — since characterization tests already pass against the existing implementation, there is nothing to implement. The skill correctly skips to the review phase. This is an important Mode 2 behavior: the whole point is to add test coverage, not to change code. The three-stage review still runs in full because even characterization tests need spec compliance review (are all exported functions covered?) and adversarial review (are the assertions meaningful or just smoke tests?).

All three units are independent (isolated utility modules), so eager dispatch immediately fills 3 of the 4 available concurrent slots. The `--skip-design` flag is redundant here (Mode 2 skips design by default) but is correctly parsed and applied. Model assignment is uniformly sonnet/high since all units are Mechanical complexity. The key risk in this mode is that characterization tests may be shallow (just calling functions without asserting on specific outputs) — the adversarial reviewer is the primary defense against this, and the assertion density check in RED verification provides an additional guardrail.
