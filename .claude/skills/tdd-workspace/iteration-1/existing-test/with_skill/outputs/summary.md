# Eval 3: Existing Test — Summary

This eval exercises Mode 3 (user-provided test), the most streamlined path through the skill. The user already has a failing test file and wants the skill to produce the implementation. The Test Writer phase and RED verification are both completely skipped, and the design gate is bypassed. The pipeline starts directly at Code Writer, runs through GREEN verification, then the full three-stage review.

The key nuance is that even though RED verification is skipped, the lead still records test file checksums before spawning the Code Writer. This ensures GREEN verification can still detect if the Code Writer tampered with the user's test file — a critical anti-cheat guardrail that applies regardless of entry mode. The lead also needs to derive a spec-contract from the test file contents (since there was no Test Writer to produce one), which it does by reading and analyzing the test file. The Code Writer receives the test file contents read from disk and the derived spec-contract, maintaining the information barrier principle even though there is no Test Writer involved.

The three-stage review (spec compliance, adversarial, code quality) runs in full strict order. This is important because even though the user wrote the tests, the implementation still needs to be verified for spec compliance, adversarial robustness (no hardcoded returns, no cheating), and code quality. The review catches issues the GREEN verification alone cannot — for example, an implementation that passes tests by hardcoding specific return values would be caught by the adversarial reviewer. This eval validates that Mode 3 correctly omits the test-writing phases while preserving all implementation-quality guarantees.
